# 선착순 쿠폰 발급 — 동시성/성능 학습 프로젝트

## 이 프로젝트의 목적 (가장 중요)
이건 "쿠폰 시스템을 완성하는" 프로젝트가 아니라 **동시성/성능 문제를 측정→진단→처방→재측정 루프로 직접 학습하는** 프로젝트다.
따라서 코드가 돌아가는 것보다 **각 단계의 병목을 숫자로 관측하고, 그 원인을 스스로 진단하는 과정**이 결과물의 핵심이다.

## AI(너)의 역할 경계 — 반드시 지킬 것
- **위임 가능**: 회원/쿠폰 정책/발급 인스턴스의 도메인 보일러플레이트, 엔티티/리포지토리 매핑, 부하 테스트 하네스(k6 등) 세팅, 모니터링 스택(Prometheus/Grafana 등) 세팅.
- **위임 불가 (사용자가 주도)**: 재고 차감 핵심 경로의 구현 방식, 락 전략 결정, 부하 시나리오 설계, 측정 결과의 해석.
- **금지**: 지시하지 않은 락/캐시/인덱스/최적화를 임의로 추가하지 말 것. 특히 재고 차감 로직은 사용자가 "이렇게 짜라"고 명시하기 전까지 **일부러 순진하게(락 없이)** 둘 것. 안전하게 미리 짜버리면 학습 대상인 버그가 사라진다.
- 코드를 제안할 때는 "왜 이렇게 되는지"를 설명하되, 결론(어느 전략이 정답인지)을 미리 확정하지 말 것. 그건 측정으로 사용자가 판단한다.

## 도메인 모델 (4개)
- **Member**: 발급 주체.
- **CouponPolicy**: 애그리거트 루트. 규칙 보유(할인 정액/정률, 선착순 수량, 발급기간, 1인 1매 여부). 거의 안 바뀌는 메타데이터.
- **CouponStock**: 재고 카운터. CouponPolicy에서 분리된 경합 지점. (단계별로 컬럼→별도테이블→Redis로 이동 예정)
- **IssuedCoupon**: 특정 멤버가 실제 발급받은 인스턴스. 상태(ISSUED→USED→EXPIRED).

관계: Member 1:N IssuedCoupon N:1 CouponPolicy. CouponStock은 CouponPolicy와 1:1.

### 두 종류의 동시성 문제 (구분해서 다룰 것)
1. **재고 카운터 쓰기 경합** → oversell. 메인 이벤트.
2. **1인 1매 중복 발급 레이스** → 같은 유저 따닥 클릭. oversell과 별개의 버그. 2단계에서 유니크 제약으로 잡고, 5단계 Redis 이전 시 함께 재설계한다. **이 스레드를 끝까지 놓치지 말 것.**

### 스코프 밖 (넣지 말 것)
Order, Payment, Product 카탈로그, 정산 등 인접 도메인. CRUD만 늘고 학습 가치 0. 쿠폰 사용(USED) API는 여력 되면 마지막에 최소한만.

## 단계별 로드맵 — 각 단계는 서로 다른 걸 가르친다
각 단계는 "가설 → 부하 측정 → 진단 → 결정"의 한 사이클이다. **결과를 미리 확정하지 말 것.** 아래 '예상'은 가설일 뿐이고, 측정이 가설을 배신하면 그게 최고의 학습 소재다.

1. **DB 컬럼 + 락 없음**: oversell과 중복발급을 눈으로 관측. (레이스가 무엇인지)
2. **DB 컬럼 + 비관락**: 정합성은 잡힘. 예상 병목=커넥션 풀 고갈로 TPS 하락. (안전 vs 처리량 트레이드오프) — 여기서 1인 1매를 유니크 제약으로 처리.
3. **재고 별도 테이블 분리 (여전히 DB 락)**: 예상=락 사정거리는 좁아지나 카운터 자체 경합은 그대로. 핵심 교훈은 "병목은 위치가 아니라 한 row 직렬화 구조"임을 측정으로 확인.
4. **낙관락 / 원자적 UPDATE** (`UPDATE ... SET cnt=cnt-1 WHERE cnt>0`): DB 안에서 락 없이 원자성 확보. **Redis 없이 여기까지 갈 수 있음을 보이는 게 포인트.**
5. **Redis로 재고 이전**: 차감을 DB 밖으로. 예상=TPS 점프. 대신 Redis-DB 정합성/유령차감/SPOF 문제 발생. (분리의 비용) 1인 1매도 여기서 재설계.

이후 여력 되면: 멀티서버 확장(단일 JVM 의존 붕괴 관측 → 분산 락/외부 상태의 필요성 체감).

### 최소 완주선 (스코프 방어)
**4단계 + 각 단계 before/after 숫자 정리**까지가 포폴 성립 최소선. 5단계와 멀티서버는 보너스. 이 프로젝트의 최빈 사망 원인은 기술이 아니라 스코프 과욕이다.

## 측정 원칙 (신뢰성 방어)
- 로컬 단일 머신 측정이므로 **절대 수치(TPS 3000!)는 무의미**. 모든 숫자는 **동일 환경에서의 상대 비교**로만 해석한다.
- 단계 간 비교 시 환경 고정(리소스 제한 건 docker-compose 등).
- 평균이 아니라 **p99 latency**와 TPS, 실패율을 함께 본다.
- "이 TPS 어디서 측정?" 질문에 "로컬이라 절대치 무의미, 상대 비교로 설계"라 답할 수 있게.
